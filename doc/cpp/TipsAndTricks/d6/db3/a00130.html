Proxy Design Pattern

Proxy Design Pattern

Въведение

Патернът Proxy е структурен шаблон за проектиране, който предоставя заместител или представител на друг обект с цел контрол на достъпа до него. Този шаблон позволява на клиентския код да взаимодейства с реалния обект чрез прокси, който може да добавя допълнителна логика като контрол на достъпа, кеширане, логване или отлагана инициализация. Proxy често се използва, когато директният достъп до обекта е скъп, рисков или нежелан поради причини, свързани със сигурността или производителността. Прокси обектът имплементира същия интерфейс като реалния обект, така че клиентът не прави разлика дали работи с прокси или с реалния обект. Това позволява прозрачно добавяне на функционалности без промяна на съществуващия код. В реални приложения Proxy се използва за контрол на достъпа до ресурси, управление на отдалечени обекти, виртуализация или мониторинг на действията върху обекти.

Пример

Ето един пример за имплементация на патерна Proxy в C++:

/**
* @file Proxy.hpp
* Demonstrates the Proxy design pattern with Subject, RealSubject, and Proxy classes for access control and logging.
* This file provides a complete example of the Proxy pattern, where the Proxy class controls access to a RealSubject
* instance and adds logging functionality. The client interacts with the Subject interface, allowing seamless
* substitution of the real object or its proxy. This pattern is useful for access control, lazy initialization,
* logging, or monitoring in embedded and UI applications.
*/
#pragma once
#include <iostream>
#include <memory>
/**
* Abstract interface for request handling.
* Defines the contract for request handling, allowing both real and proxy implementations to be used interchangeably
* by client code. This interface enables transparent substitution of real objects and proxies, facilitating extension
* and control of access or behavior.
*/
class Subject {
public:
/**
* Handles a request operation.
* @pre Must be implemented by derived classes to provide specific request handling logic.
*/
virtual void request() const = 0;
/**
* Default virtual destructor for Subject.
* @pre Ensures proper cleanup of derived objects when deleted via base pointer.
*/
virtual ~Subject() = default;
};
/**
* Concrete implementation of Subject for real request handling.
* Provides the actual business logic for handling requests. This class is used by the Proxy to delegate real work,
* and can be used directly by clients when no proxy is needed.
*/
class RealSubject : public Subject {
public:
/**
* Handles the request by executing the real subject's logic.
* @pre Called by client or proxy to perform the actual operation.
*/
void request() const override { std::cout << "RealSubject: Handling request.\n"; }
};
/**
* Proxy implementation of Subject for access control and logging.
* Controls access to a RealSubject instance and adds logging functionality. The Proxy checks access before delegating
* the request to the real subject and logs the operation after execution. This enables additional behavior without
* modifying the real subject or client code.
*/
class Proxy : public Subject {
private:
std::unique_ptr<RealSubject> realSubject; ///< Unique pointer to the real subject instance.
public:
/**
* Constructs a Proxy with a new RealSubject instance.
* @pre Initializes the proxy with exclusive ownership of a RealSubject.
*/
Proxy() : realSubject(std::make_unique<RealSubject>()) {}
/**
* Handles the request by checking access, delegating to the real subject, and logging the operation.
* @pre Access is checked before delegating to the real subject; logging occurs after the request.
*/
void request() const override {
if (checkAccess()) {
realSubject->request();
logAccess();
}
}
/**
* Checks access before delegating the request to the real subject.
* @return Returns true if access is granted; otherwise, false.
* @pre Called before forwarding the request to the real subject.
*/
bool checkAccess() const {
std::cout << "Proxy: Checking access prior to firing a real request.\n";
return true;
}
/**
* Logs the access after the request has been handled.
* @pre Called after the real subject has processed the request.
*/
void logAccess() const { std::cout << "Proxy: Logging the time of request.\n"; }
};
/**
* Invokes the request method on a Subject instance.
* @param subject Reference to a Subject object (real or proxy).
* @pre The subject must implement the request method.
*/
void clientCode(const Subject & subject) { subject.request(); }
/**
* Entry point demonstrating usage of RealSubject and Proxy.
* @return Returns 0 on successful execution.
* @pre Demonstrates both direct and proxy-based request handling.
*/
int main() {
std::cout << "Client: Executing the client code with a real subject:\n";
RealSubject realSubject;
clientCode(realSubject);
std::cout << "\nClient: Executing the same client code with a proxy:\n";
Proxy proxy;
clientCode(proxy);
return 0;
}
Подробно описание на примера:
В този пример класът Subject дефинира общ интерфейс с виртуален метод request, който трябва да бъде реализиран от всички наследници. Класът RealSubject предоставя реалната имплементация на този метод и изпълнява основната бизнес логика. Класът Proxy също наследява от Subject и съдържа уникален указател към обект от тип RealSubject. При извикване на метода request през прокси, първо се извършва проверка за достъп чрез метода checkAccess, след което, ако достъпът е разрешен, се делегира изпълнението към реалния обект. След изпълнението се извиква методът logAccess за логване на действието. По този начин прокси обектът може да добавя допълнителна функционалност (контрол на достъпа, логване) без промяна на реалния обект. В main функцията се демонстрира използването както на реалния обект, така и на прокси, като клиентският код работи с интерфейса Subject и не прави разлика между реалния обект и проксито. Това осигурява гъвкавост и възможност за разширяване на функционалността без промяна на клиентския код или реалната имплементация.
