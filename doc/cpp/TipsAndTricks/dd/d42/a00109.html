Adapter Pattern

Въведение

Шаблонът Adapter позволява на обекти с несъвместими интерфейси да работят заедно. Той действа като мост между два несъвместими интерфейса, като преобразува интерфейса на един клас в интерфейс, очакван от клиентите. Това позволява класове с различни интерфейси да си сътрудничат без модификация. Шаблонът Adapter е структурен дизайн шаблон, който улеснява интеграцията на класове с несъвместими интерфейси без необходимост от промени в съществуващия код. Adapter капсулира обект с различен интерфейс и предоставя необходимите методи, така че клиентският код да може да използва този обект, сякаш е от очаквания тип. Това е особено полезно при работа с външни библиотеки, наследен код или при миграция към нови интерфейси в големи системи. Adapter може да бъде имплементиран чрез наследяване или композиция, като композицията често се предпочита за по-голяма гъвкавост. Основната цел е да се осигури съвместимост между класовете, без да се нарушава принципът open/closed и без да се въвеждат промени в съществуващите класове.

Пример

Следва примерна имплементация на шаблона Adapter на C++:

/**
* @file Adapter.hpp
* Demonstrates the Adapter design pattern in C++ for bridging incompatible interfaces.
* This file provides a sample implementation of the Adapter pattern, allowing objects with incompatible
* interfaces to collaborate by introducing an adapter class that translates requests from the expected
* interface to the adaptee's interface. The example showcases how to use inheritance and composition to
* achieve interface compatibility without modifying existing code, which is particularly useful in embedded
* systems and legacy integration scenarios.
*/
#pragma once
#include <iostream>
/**
* Represents the target interface expected by clients.
* This class defines the interface that client code expects to use, enabling polymorphic behavior and
* allowing adapters to provide compatibility with other interfaces by inheriting from this class.
*/
class ClientInterface {
public:
/**
* Handles the default request using the client interface.
* @return None.
*/
virtual void handleRequest() const { std::cout << "ClientInterface: The default behavior." << std::endl; }
};
/**
* Represents an existing class with an incompatible interface.
* This class provides a specific interface that is not compatible with the client interface, requiring an
* adapter to enable interoperability with client code expecting the client interface.
*/
class LegacyService {
public:
/**
* Handles a specific request using the legacy service's interface.
* @return None.
*/
void performLegacyOperation() const { std::cout << "LegacyService: Legacy operation." << std::endl; }
};
/**
* Adapts the LegacyService interface to the ClientInterface.
* This class implements the client interface and internally uses a legacy service instance to translate
* requests, enabling client code to interact with legacy service objects as if they were client interfaces
* without modifying either the client or legacy service code.
*/
class LegacyAdapter : public ClientInterface {
private:
LegacyService * legacyService; ///< Pointer to the legacy service instance used for request translation.
public:
/**
* Constructs a LegacyAdapter with the given legacy service instance.
* @param legacyService Pointer to the legacy service object to be adapted.
*/
LegacyAdapter(LegacyService * legacyService) : legacyService(legacyService) {}
/**
* Handles the request by translating it to the legacy service's operation.
* @return None.
* @see LegacyService::performLegacyOperation
*/
void handleRequest() const override { legacyService->performLegacyOperation(); }
};
/**
* Entry point demonstrating the Adapter pattern usage.
* @return Exit status code.
*/
int main() {
LegacyService * legacyService = new LegacyService;
ClientInterface * client = new LegacyAdapter(legacyService);
client->handleRequest();
delete legacyService;
delete client;
return 0;
}
Подробно описание на примера:
Класът Target предоставя виртуален метод, който клиентският код очаква да използва. Класът Adaptee реализира функционалност, която клиентът иска да използва, но чрез различен метод с несъвместим интерфейс. Класът Adapter наследява от Target и съдържа инстанция на Adaptee (композиция). В имплементацията на метода на Target, Adapter извиква съответния метод на Adaptee, като по този начин преобразува интерфейса. Това позволява на клиентския код да работи с обекти от тип Target, без да се интересува от детайлите на реализацията на Adaptee. Този подход улеснява интеграцията на съществуващ или външен код, без да се налага промяна на оригиналните класове, като същевременно се спазват принципите на обектно-ориентираното програмиране и се осигурява по-голяма гъвкавост и разширяемост на системата.
